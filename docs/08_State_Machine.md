# 桌面互动软件 - 08_核心控制流与状态机文档

## 1. 模块概述
本模块是整个桌面互动软件的“交通指挥中心（FSM，有限状态机）”。它统筹管理视觉抓取、API 请求、语音播报和麦克风监听等各个子模块的运行生命周期。通过严格的互斥锁和打断机制，本模块确保挂件在复杂的桌面环境（如多开游戏、频繁切屏）中依然保持逻辑清晰，做到“不抢话、不乱看、懂进退”。

## 2. 核心状态枚举 (Core States)
系统在任意时刻仅能处于以下五种主状态之一：

1. **`IDLE` (静默待机)**：挂件处于半透明退让状态，底层 02 模块的截图倒计时正常运行，等待事件触发。
2. **`OBSERVING` (视觉分析中)**：已抓取有效屏幕截图，正在向大模型 API 发起请求，等待返回结果。
3. **`SPEAKING` (表达播报中)**：API 已返回，挂件实体化，正在进行文本气泡展示与 TTS 语音播报。
4. **`LISTENING` (专注倾听中)**：语音播报完毕后的 10 秒交互窗口期，麦克风开启，等待用户语音输入。
5. **`PROCESSING_AUDIO` (语音思考中)**：捕获到用户语音并转写完毕，正在向大模型请求对话回复。

## 3. 核心交通规则：状态流转与互斥逻辑
为了打造极致的陪伴体验且绝对不干扰用户操作，状态机强制执行以下三条核心规则：

### 3.1 规则一：短时切屏免疫 (3秒驻留判定)
* **应用场景**：用户游戏双开（如《明日方舟》挂机与《英雄联盟》匹配同时进行），或快速按 `Alt+Tab` 切歌/查攻略。
* **执行逻辑**：当 02 模块检测到窗口切换事件时，**不立即截图，而是启动一个 3 秒的“驻留确认计时器”**。
  * 如果在 3 秒内，用户再次切换了窗口，则判定为“无效的快速切屏”，直接取消本次截图计划。
  * 只有当用户在新窗口**持续停留超过 3 秒**时，才正式触发截图并进入 `OBSERVING` 状态。
  * **旧请求丢弃**：如果在旧的 API 请求还未返回时，用户又触发了新的有效截图，状态机将直接丢弃旧的 API 响应，确保挂件永远只对“眼前最新的画面”做出反应。

### 3.2 规则二：活跃期全局冻结 (交互原子性)
* **应用场景**：防止挂件在说话或倾听时，后台定时器依然在盲目截图，导致逻辑冲突。
* **执行逻辑**：当系统处于 **活跃状态**（即 `OBSERVING`、`SPEAKING`、`LISTENING`、`PROCESSING_AUDIO`）时，向 02 模块下发“冻结指令”。
  * 彻底暂停“切屏事件监听”与“5分钟定时兜底倒计时”。
  * 直到当前这一轮完整的交互闭环结束，系统重新回落至 `IDLE` 状态后，02 模块的监听器和倒计时器才会被重置并重新启动。

### 3.3 规则三：物理级强力打断 (摇晃闭嘴机制)
* **应用场景**：挂件当前的话题用户不感兴趣，或者突然需要安静的环境。
* **执行逻辑**：提供最高优先级的物理中断。系统实时监听前端窗口区域内的鼠标轨迹变化率。
  * **触发条件**：用户将鼠标指针移至挂件 UI 区域上方，并进行**快速来回晃动**。
  * **处理结果**：无视当前处于任何状态，状态机瞬间发出 `KILL` 指令。立即掐断 TTS 音频流、清空文字气泡、取消所有排队中的 API 请求，并强制将状态重置为 `IDLE`（休眠）。

## 4. 异常状态恢复 (Error Recovery)
* **API 超时/崩溃**：如果在 `OBSERVING` 或 `PROCESSING_AUDIO` 状态下停留超过 15 秒未收到大模型响应，状态机自动触发超时机制。放弃本次请求，播放简短的提示音或微表情（表示“走神了”），随后安全回落至 `IDLE`。
